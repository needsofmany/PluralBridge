The access path is:

Account
→ active membership
→ active system

In NT terms:

Account = security principal
Membership = access relationship / authorization grant
Role = constrained permission class
System = object namespace / security boundary
System data = protected objects inside that namespace

So Account is the login identity. It reaches PluralBridge data through Membership. System is the data container and boundary. Everything like members, front history, privacy buckets, and source records hangs under SystemId.

A membership says this Account is connected to this System.

A role says what that Account is allowed to do inside that System.

In NT/security terms:

Account = principal / login identity
System = protected object namespace / data boundary
Membership = this principal has an access relationship to this namespace
Role = permission bundle attached to that relationship

So the membership is the relationship record. The role is authority on that relationship.

Example from your current /api/me:

Account:
demo@thepluralbridge.local

System:
826d77cf-8b1a-a301-4efe-1113e5a17e88

Membership:
demo@thepluralbridge.local is an Active member of that System

Role:
Owner

Meaning:

This account belongs to this system, and within that system it has Owner authority.

The important distinction is that a role does not stand alone. Saying:

demo@thepluralbridge.local has Owner

is incomplete. Owner of what? The answer is provided by the membership:

demo@thepluralbridge.local has Owner role through its membership in System X.

Some examples:

Account A -> System 1 -> Active membership -> Owner role

Account A can manage System 1.

Account B -> System 1 -> Active membership -> Editor role

Account B can edit data in System 1, but may not manage membership or system settings.

Account C -> System 1 -> Active membership -> Viewer role

Account C can read allowed data in System 1, but cannot edit.

Account A -> System 2 -> Active membership -> Viewer role

The same login account can have a different role in a different system.

Account D -> System 1 -> Suspended membership -> Owner role

Even if a role row exists, the membership status blocks access because the membership is not active.

So the access check order should be:

1. Who is the account?
2. Does that account have an active membership in this system?
3. What roles are attached to that active membership?
4. Does one of those roles allow the requested operation?

For PluralBridge, that means:

Membership answers: are you connected to this plural system at all?
Role answers: what can you do there?

That is why the current table shape is correct:

pb_system_memberships

holds the account/system relationship.

pb_system_membership_roles

attaches one or more roles to that specific relationship.

pb_roles

defines the reusable role names like Owner, Admin, Editor, Viewer, etc.
