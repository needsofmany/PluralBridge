My recommendation for PluralBridge:

Account
└── Membership(s)
    └── System
        └── Subsystem(s)
            └── Subsystem(s)

The membership should attach to the System security boundary, then subsystem access can be derived from that boundary unless a subsystem deliberately declares its own boundary.

So the rule should be:

If a subsystem is part of the same plural system boundary:
    it inherits the parent System membership.

If a subsystem needs different owners, roles, visibility, consent, or audit authority:
    it needs its own membership boundary.

For Chapter 2, I would keep the model simple:

pb_accounts
pb_systems
pb_memberships

Then later, subsystem hierarchy can fit inside pb_systems with something like:

pb_systems
├── SystemId
├── ParentSystemId nullable
├── SystemName
└── ...

That gives you this shape:

Account
└── Membership → Root System
                  ├── Child System A
                  ├── Child System B
                  └── Child System C

The important design point: membership inheritance should be an explicit rule, not an accident of the parent-child structure.

In NT terms:

Account = principal
Membership = access token grant
System = securable namespace
Subsystem = child container
ParentSystemId = namespace hierarchy
Inherited access = ACL inheritance rule

So for now: yes, subsystems normally share the parent system membership.

Later: allow override membership only when a subsystem becomes its own security boundary.

pb_systems remains the entity table. The hierarchy lives beside it.

I would call the new table an adjacency table rather than a join table, because each row says “this system has this parent.” Same practical result, cleaner mental model.

pb_systems
└── one row per system/subsystem

pb_system_relationships
└── one row per system's place in the hierarchy

Suggested minimum shape:

pb_system_relationships

SystemId          uniqueidentifier not null
ParentSystemId    uniqueidentifier null
SystemLevel       int not null

Rows:

Root:
SystemId          = root-sys-id
ParentSystemId    = NULL
SystemLevel       = 1

Child:
SystemId          = child-sys-id
ParentSystemId    = root-sys-id
SystemLevel       = 2

Grandchild:
SystemId          = grandchild-sys-id
ParentSystemId    = child-sys-id
SystemLevel       = 3

I would avoid the name Rank for that column. In database terms, Rank sounds like sibling ordering or display priority. Your value is tree depth, so SystemLevel, Depth, or TreeLevel is clearer.

The key rule:

Every system row has exactly one hierarchy row.
The root hierarchy row has ParentSystemId = NULL.
A child hierarchy row points to its parent system.

For constraints:

PK or unique key on SystemId
FK SystemId → pb_systems.SystemId
FK ParentSystemId → pb_systems.SystemId
CHECK ParentSystemId IS NULL OR ParentSystemId <> SystemId
CHECK SystemLevel >= 1

This keeps Chapter 2 stable:

Account
→ Membership
→ Root System
→ System hierarchy

Membership can still attach to the root system boundary. Subsystems inherit through pb_system_relationships.

For access resolution later:

Requested SystemId
→ walk parent chain to root
→ verify Account has active Membership on root
→ allow access to requested subsystem data

So yes: your proposed direction is sound. I would name the table pb_system_relationships or pb_system_hierarchy, use SystemLevel instead of Rank, and require one row for the root with ParentSystemId = NULL.
